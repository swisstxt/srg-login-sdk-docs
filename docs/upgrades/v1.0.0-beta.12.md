# Upgrade to v1.0.0-beta.12

> **Released**: May 2026 · **Codename**: "Migration & Identity"
> **Estimated effort**: ~30–45 min per app

This release brings four major new features, the last batch of breaking changes before the 1.0 release candidate, and several stability improvements.

:::info Bundled migration effort
All breaking changes are concentrated in this beta. After upgrading, the next SDK versions (RC.1 and 1.0.0 stable) are not expected to introduce new breaking changes unless a security issue forces it.
:::

## At a glance

| | |
|---|---|
| 🆕 New features | 4 (Session migration, Token refresh modes, Structured ID Token claims, tvOS AuthorizationCodeFlow) |
| ⚠️ Breaking changes | 7 (grouped on 2 tickets: LOGIN-1825 and LOGIN-1858) |
| 🐛 Critical bug fixes | 1 (JWT Base64URL parsing) |
| 🏗️ Internal refactor | Pluggable login strategies (no consumer migration needed) |
| 🎯 Estimated effort per app | ~30–45 min |

## What's new

### 🆕 Session migration via `LoginMethod.Migration` (LOGIN-1846)

Apps switching from a previous SDK or a custom OAuth integration can hand over their existing refresh token to the SRG Login SDK. The SDK calls the IdP's token endpoint, obtains a fully validated token set, and seeds its secure storage — without forcing the user to re-authenticate.

<Tabs>
  <TabItem value="android" label="Android" default>

```kotlin
// On first launch after the app is updated to the SRG Login SDK
val oldRefreshToken = oldSdkStorage.readRefreshToken()

srgLogin.login(
    loginMethod = LoginMethod.Migration(refreshToken = oldRefreshToken),
    authContext = context,
).collect { state ->
    when (state) {
        is LoginState.Success -> {
            oldSdkStorage.clear()
            // user remains signed in → navigate to main screen
        }
        is LoginState.Failure -> {
            // refresh token revoked / expired → fall back to a regular login
        }
        else -> { /* intermediate states not applicable here */ }
    }
}
```

  </TabItem>
  <TabItem value="ios" label="iOS">

```swift
// On first launch after the app is updated to the SRG Login SDK
let oldRefreshToken = oldSdkStorage.readRefreshToken()

let states = srgLogin.login(
    loginMethod: .migration(refreshToken: oldRefreshToken),
    authContext: authContext
)

for await state in states {
    switch state {
    case let success as LoginStateSuccess:
        oldSdkStorage.clear()
        // user remains signed in → navigate to main screen
    case is LoginStateFailure:
        // refresh token revoked / expired → fall back to a regular login
        break
    default:
        break // intermediate states not applicable here
    }
}
```

  </TabItem>
</Tabs>

**Prerequisites**: a valid refresh token from the previous session, and the same `clientId` and IdP between the old integration and the SRG Login SDK.

Use `LoginMethod.Migration` exactly **once**, on the first launch after the app upgrade introducing the SDK.

### 🆕 Token refresh modes — `TokenRefreshMode` (LOGIN-1825)

The SDK now exposes two clear modes for managing token refresh:

| Mode | For whom | Behavior |
|---|---|---|
| `TokenRefreshMode.Automatic` *(default)* | 95% of apps | The SDK monitors token state and refreshes automatically before expiration |
| `TokenRefreshMode.Manual` | Apps that need to control refresh timing | No background monitoring; the app refreshes on-demand via `getAccessToken(forceRefresh = true)` |

<Tabs>
  <TabItem value="android" label="Android" default>

```kotlin
val config = SrgLoginConfig(
    environment = Environment.PROD,
    clientId = "...",
    appIdentity = appIdentity,
    tokenRefreshConfig = TokenRefreshConfig(
        mode = TokenRefreshMode.Automatic,  // or TokenRefreshMode.Manual
    ),
)
```

  </TabItem>
  <TabItem value="ios" label="iOS">

```swift
let config = SrgLoginConfig(
    environment: .prod,
    clientId: "...",
    appIdentity: appIdentity,
    tokenRefreshConfig: TokenRefreshConfig(
        mode: TokenRefreshModeAutomatic.shared  // or TokenRefreshModeManual.shared
    )
)
```

  </TabItem>
</Tabs>

This replaces the previous configuration with two boolean flags and an unused `monitoringIntervalSeconds` field. See [Breaking changes](#breaking-changes) for the migration table.

### 🆕 Structured ID Token claims — `TokenSet.idTokenClaims` (LOGIN-1858)

After a successful login, the user's identity claims (name, email, picture, locale, address, etc.) are exposed as a typed, OIDC-compliant object — no more manual JWT parsing on the consumer side.

<Tabs>
  <TabItem value="android" label="Android" default>

```kotlin
// Inside your login state collector, after a successful login
if (state is LoginState.Success) {
    val claims = state.tokenSet.idTokenClaims
    val displayName = claims?.defaultDisplayName()  // smart fallback chain
    val email       = claims?.email
    val picture     = claims?.picture
    val locale      = claims?.locale
    val address     = claims?.address  // structured: street, city, postalCode, country
}
```

  </TabItem>
  <TabItem value="ios" label="iOS">

```swift
// Inside your login state collector, after a successful login
if let success = state as? LoginStateSuccess {
    let claims = success.tokenSet.idTokenClaims
    let displayName = claims?.defaultDisplayName()   // smart fallback chain
    let email       = claims?.email
    let picture     = claims?.picture
    let locale      = claims?.locale
    let address     = claims?.address                // structured: street, city, postalCode, country
}
```

  </TabItem>
</Tabs>

`defaultDisplayName()` returns the best available name with a fallback chain: `name` → `givenName + familyName` → `preferredUsername` → `email` → `subject`.

:::caution Claims depend on the requested scopes
The properties available in `idTokenClaims` are conditioned by the OAuth/OIDC scopes requested at `login()` time (per OIDC Core §5.4):

| Scope requested | Claims available |
|---|---|
| `openid` *(always required for OIDC)* | `subject`, `issuer`, `audience`, `expiration`, `nonce`, `authTime` |
| `profile` | `name`, `givenName`, `familyName`, `preferredUsername`, `picture`, `locale`, `gender`, `birthdate`, `updatedAt`, … |
| `email` | `email`, `emailVerified` |
| `phone` | `phoneNumber`, `phoneNumberVerified` |
| `address` | `address` (street, city, postal code, country) |

Adjust your `login()` call to request the scopes your app needs:

```kotlin
srgLogin.login(
    LoginMethod.Web,
    authContext,
    additionalScopes = listOf("profile", "email"),
)
```
:::

The full API also includes typed accessors for custom claims: `getStringClaim()`, `getLongClaim()`, `getBooleanClaim()`, `getStringListClaim()`, `getJsonObjectClaim()`, `hasClaim()` — all with strict JSON type checks.

### 🆕 tvOS — Authorization Code Flow via `ASWebAuthenticationSession` (LOGIN-1832)

tvOS apps can now use the standard browser-based authorization code flow (PKCE) via Apple's `ASWebAuthenticationSession`, in addition to the existing Device Authorization Grant (RFC 8628).

`LoginMethod.Web` now works on tvOS with the same API as iOS. No app-side change is required beyond targeting tvOS.

```swift
// Swift, on tvOS
for await state in srgLogin.login(loginMethod: .web, authContext: ctx) {
    switch state {
    case let success as LoginStateSuccess: // ...
    case let failure as LoginStateFailure: // ...
    default: break
    }
}
```

## ⚠️ Breaking changes

All breaking changes are grouped on **two tickets** so the migration is a single coordinated effort.

| # | Ticket | Before | After | Action |
|---|---|---|---|---|
| 1 | LOGIN-1825 | `srgLogin.refreshToken()` | `srgLogin.getAccessToken(forceRefresh = true)` | Rename one call site |
| 2 | LOGIN-1825 | `TokenRefreshConfig(enableProactiveRefresh = true, enableAutomaticTokenMonitoring = true)` | `TokenRefreshConfig(mode = TokenRefreshMode.Automatic)` | Adjust your `SrgLoginConfig` initialization |
| 3 | LOGIN-1825 | `monitoringIntervalSeconds = 60` | *(removed — computed automatically from token lifetime)* | Remove the parameter from `TokenRefreshConfig` |
| 4 | LOGIN-1825 | `TokenState.Refreshed(tokenSet)` carries a `TokenSet` payload | `TokenState.Refreshed` is now a `data object` *(no payload)* | Read the token via `getAccessToken()` instead of through the state |
| 5 | LOGIN-1825 | `observeTokenState()` initial value is `TokenState.NoTokens` | Initial value is `TokenState.Uninitialized` | Add an `Uninitialized` branch in your `when` (treat as "show splash, do not navigate to login yet") |
| 6 | LOGIN-1858 | `TokenSet.scope: String?` (space-separated) | `TokenSet.scopes: List<String>` | Replace `tokenSet.scope?.split(" ")?.contains("openid")` with `tokenSet.hasScope("openid")` (or use `tokenSet.scopes` directly) |
| 7 | LOGIN-1858 | `AccessToken(value, expiresAt, expiresIn, isExpired, issuer, subject, issuedAt)` | `AccessToken(value, expiresAt, issuer, subject, issuedAt)` *(constructor reduced from 7 to 5 params; `expiresIn` and `isExpired` are now computed properties)* | Most apps use `AccessToken.fromJwt()` (factory) and don't call the constructor directly. If you do, adjust the call. |

## Stability improvements *(no action required)*

- **`TokenState` no longer gets stuck at `Refreshing`** during concurrent refreshes, even when the IdP responds with `invalid_grant` mid-refresh (LOGIN-1846)
- **`observeTokenState()` now works correctly in Manual mode** — it emits state on every `getAccessToken()` call instead of staying silent (LOGIN-1825)
- **No more "flash" of the login screen at startup** when the user has a valid persisted session — the new `Uninitialized` initial state lets you show a splash screen until the SDK has evaluated storage (LOGIN-1825)
- **JWT parser fixed for Base64URL alphabet** — login no longer fails for users whose JWT contains `-` or `_` characters, observed in production (LOGIN-1848)

## Per-platform notes

### Android

#### Trust anchor on Android 7.0 / 7.1.0

Devices running Android 7.0 or 7.1.0 may hit a TLS error on the **INT** environment because the Let's Encrypt root certificate (ISRG Root X1) is absent from those devices' trust store, and the IdenTrust cross-sign that previously bridged this gap expired in September 2021.

If your app supports Android 7.x:

1. Bundle ISRG Root X1 as a raw resource: `res/raw/isrg_root_x1.pem` — download from [letsencrypt.org/certs/isrgrootx1.pem](https://letsencrypt.org/certs/isrgrootx1.pem)

2. Add a Network Security Config:

   ```xml
   <!-- res/xml/network_security_config.xml -->
   <?xml version="1.0" encoding="utf-8"?>
   <network-security-config>
       <base-config>
           <trust-anchors>
               <certificates src="system" />
               <certificates src="@raw/isrg_root_x1" />
           </trust-anchors>
       </base-config>
   </network-security-config>
   ```

3. Reference it in your `AndroidManifest.xml`:

   ```xml
   <application android:networkSecurityConfig="@xml/network_security_config">
   ```

:::warning Use `<base-config>`, not `<domain-config>`
The SDK uses Ktor's CIO HTTP engine on Android, which is incompatible with Android `<domain-config>` blocks (they require a hostname-aware `checkServerTrusted` that CIO does not call — [KTOR-3540](https://youtrack.jetbrains.com/issue/KTOR-3540)). `<base-config>` works correctly. Adding ISRG Root X1 globally is safe — it is a publicly trusted CA already present on Android 7.1.1+; we are restoring the trust state that newer devices already have.
:::

#### Min SDK

The SDK supports `minSdkVersion = 21` (Android 5.0). No change in this release.

### iOS

No specific iOS configuration is required for this release.

#### Swift API access

Thanks to the SKIE plugin, all Kotlin types — `LoginState`, `TokenSet`, `IdTokenClaims`, etc. — are available natively in Swift with idiomatic naming. Kotlin `Flow<T>` is exposed as Swift `AsyncSequence`, and Kotlin `suspend fun` is exposed as Swift `async fun`.

```swift
let states = srgLogin.login(loginMethod: .web, authContext: ctx)
for await state in states {
    switch state {
    case let success as LoginStateSuccess:
        let claims = success.tokenSet.idTokenClaims
        // ...
    default: break
    }
}
```

## Migration checklist

Use this checklist to track your migration:

- [ ] Bumped SDK dependency to `1.0.0-beta.12`
- [ ] Replaced `srgLogin.refreshToken()` with `srgLogin.getAccessToken(forceRefresh = true)`
- [ ] Replaced `TokenRefreshConfig(enableProactiveRefresh, enableAutomaticTokenMonitoring, monitoringIntervalSeconds)` with `TokenRefreshConfig(mode = TokenRefreshMode.Automatic)` (or `Manual` if relevant)
- [ ] Removed `TokenState.Refreshed.tokenSet` access; switched to `getAccessToken()` for the value
- [ ] Added `TokenState.Uninitialized` branch in observers (typically: show splash)
- [ ] Replaced `tokenSet.scope?.split(...)` with `tokenSet.hasScope(...)` or `tokenSet.scopes`
- [ ] Updated `AccessToken(...)` constructor calls if any (most apps don't call it directly)
- [ ] *(Android only)* Added Network Security Config for Android 7.x trust anchor if your minSdk includes Android 7.x
- [ ] Reviewed the new `LoginMethod.Migration` if your app is migrating users from a previous integration
- [ ] If you display user info, considered using the new `tokenSet.idTokenClaims` instead of parsing the ID token manually
- [ ] Verified that `additionalScopes` includes the scopes your app needs (`profile`, `email`, etc.) for the claims you read
- [ ] Smoke-tested login, token refresh, and logout end-to-end on Android and iOS
- [ ] Reviewed your app's CI to confirm tests still pass after the dependency bump
